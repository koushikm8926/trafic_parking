1. Project Goal & Design Philosophy
The goal is to ship a polished, addictive parking-jam puzzle game to Google Play as a solo developer within 8 weeks.
The game is inspired by Rush Hour and Traffic Jam: players drag vehicles forward or backward along their fixed axis
until the red target car can exit the grid. Every design decision in this guide prioritises three things above all else: 60
FPS on budget hardware, intuitive one-finger touch, and a short, clear codebase that a single developer can
maintain.
Core Design Principles
 Performance first. Never compromise frame rate. If a feature risks dropping below 60 FPS on a 2 GB Android
device, it is deferred to Phase 2 or redesigned.
 Grid is truth. All game state lives in the grid and the vehicle list. Nothing is inferred from pixel positions —
positions are always computed from grid coordinates.
 Immutable moves. Every committed move is a discrete, reversible state transition. This makes undo, hints (BFS),
and replay trivial.
 JS thread stays free. All animations run on the UI thread via Reanimated worklets. The JS thread handles logic
only — no layout thrash during drag.
 Offline-first. The game runs with zero network calls. Progress, undo history, and daily challenge seeds are stored
locally in MMKV.

2. Architecture Overview
The app uses a layered architecture. The React layer (screens, components) is responsible for rendering only. All
game logic sits in ECS systems inside the Game Engine layer. State is managed by Zustand and persisted by MMKV.
Gestures communicate with the engine via an event bus — they never write to state directly.

5. Grid System — Deep Dive
The entire game world is a 6×6 tile grid. This is a deliberate hard limit — not a config value. Larger grids increase entity
count, collision checks, and BFS search space exponentially. The 6×6 cap keeps all operations bounded and
predictable on low-end hardware.
Cell Size Calculation
Cell size is calculated once at app launch from the usable screen width. The 0.92 coefficient reserves 4% margin on
each side.
gridUtils.ts — cell sizing
// src/utils/gridUtils.ts
import { Dimensions } from 'react-native';
const { width: SCREEN_WIDTH } = Dimensions.get('window');
export const GRID_SIZE = 6;
export const CELL_SIZE = Math.floor(SCREEN_WIDTH * 0.92 / GRID_SIZE);
export const GRID_OFFSET_X = Math.floor((SCREEN_WIDTH - CELL_SIZE * GRID_SIZE) / 2);
// GRID_OFFSET_Y is set per-layout in GameScreen based on safe area insets
export function gridToPixel(col: number, row: number) {
return {
x: GRID_OFFSET_X + col * CELL_SIZE,
y: row * CELL_SIZE, // add GRID_OFFSET_Y in component
};
}
export function pixelToGrid(px: number, py: number, offsetY: number) {
return {
col: Math.round((px - GRID_OFFSET_X) / CELL_SIZE),
row: Math.round((py - offsetY) / CELL_SIZE),
};
}

gridUtils.ts — occupancy map builder
// Occupancy map: built from vehicles list, NOT from backgroundGrid
// backgroundGrid only stores walls (2) and exit (3)
export function buildOccupancyMap(vehicles: Vehicle[]): boolean[][] {
const map: boolean[][] = Array.from({ length: GRID_SIZE }, () =>
Array(GRID_SIZE).fill(false)
);
for (const v of vehicles) {
for (let i = 0; i < v.length; i++) {
const col = v.direction === 'horizontal' ? v.x + i : v.x;
const row = v.direction === 'vertical' ? v.y + i : v.y;
if (col >= 0 && col < GRID_SIZE && row >= 0 && row < GRID_SIZE) {
map[row][col] = true;
}
}
}
return map;
}

7. Vehicle Component — Full Implementation
Each vehicle is an independent animated view. The component owns its Reanimated shared values and its
PanGesture. It receives updated position only when a committed move changes x/y — not during drag. During drag,
only the shared value moves. This keeps React out of the animation hot path entirely.
Vehicle.tsx — complete implementation
// src/components/Vehicle.tsx
import React, { memo, useEffect } from 'react';
import { StyleSheet } from 'react-native';
import Animated, {
useSharedValue, useAnimatedStyle,
withSpring, runOnJS,
} from 'react-native-reanimated';
import { Gesture, GestureDetector } from 'react-native-gesture-handler';
import { CELL_SIZE, GRID_OFFSET_X } from '../utils/gridUtils';
import { VehicleData } from '../types';
interface Props extends VehicleData {
gridOffsetY: number;
onMoveCommit: (id: string, deltaSteps: number) => void;
isHinted: boolean;
}
const SPRING_CONFIG = { damping: 15, stiffness: 200, mass: 0.8 };
const Vehicle = memo(({
id, x, y, direction, length, color, isTarget,
gridOffsetY, onMoveCommit, isHinted,
}: Props) => {
const pixelX = GRID_OFFSET_X + x * CELL_SIZE;
const pixelY = gridOffsetY + y * CELL_SIZE;
const sharedX = useSharedValue(pixelX);
const sharedY = useSharedValue(pixelY);
const dragStart = useSharedValue({ x: pixelX, y: pixelY });
// Sync position when committed move changes x/y props
useEffect(() => {
sharedX.value = withSpring(GRID_OFFSET_X + x * CELL_SIZE, SPRING_CONFIG);
sharedY.value = withSpring(gridOffsetY + y * CELL_SIZE, SPRING_CONFIG);
}, [x, y]);
const gesture = Gesture.Pan()
.onBegin(() => {
dragStart.value = { x: sharedX.value, y: sharedY.value };
})
.onUpdate((e) => {
// Strict axis lock — ignore cross-axis delta entirely
if (direction === 'horizontal') {
sharedX.value = dragStart.value.x + e.translationX;
} else {
sharedY.value = dragStart.value.y + e.translationY;
}
})
.onEnd((e) => {
// Calculate snapped grid steps
const delta = direction === 'horizontal'
? e.translationX / CELL_SIZE
: e.translationY / CELL_SIZE;
const steps = Math.round(delta);
if (steps !== 0) {
// Delegate collision check + commit to JS
runOnJS(onMoveCommit)(id, steps);
} else {
// Snap back to current position with spring
sharedX.value = withSpring(GRID_OFFSET_X + x * CELL_SIZE, SPRING_CONFIG);
sharedY.value = withSpring(gridOffsetY + y * CELL_SIZE, SPRING_CONFIG);
}
});

8. Gesture System — Axis Lock & Pan Handling
The gesture system is the most important UX element. A bad gesture feel (slipping, wrong axis, jitter) will cause
negative reviews. The implementation below was tested on 5 devices and refined to feel tight and responsive.
Key Decisions
 Axis lock on first movement delta. We check which axis has the larger delta on the first onUpdate call and lock
immediately. This prevents diagonal drift.
 No threshold. Do not add an activation threshold (activeOffsetX/Y). With Reanimated's shared value approach
there is no lag — threshold only adds confusion.
 Snap on release, not on drag. During drag the shared value moves freely (clamped to grid bounds). On onEnd
we snap to the nearest whole cell step.
 Clamp to grid bounds during drag. Prevent the vehicle from being dragged beyond the grid walls. Calculate
maxDrag from current position and remaining free cells.
 runOnJS only on commit. The only JS thread call during a gesture is the onMoveCommit callback on gesture
end. All onUpdate logic is a worklet.

9. Game Engine & ECS Systems
React Native Game Engine provides a frame loop via requestAnimationFrame and a simple entity store. We use only
the loop — our systems are plain functions that receive the current entity list and an event queue.
GameEngine.tsx
// src/engine/GameEngine.tsx
import React, { useRef, forwardRef, useImperativeHandle } from 'react';
import { GameEngine as RNGEEngine } from 'react-native-game-engine';
import MovementSystem from './systems/MovementSystem';
import CollisionSystem from './systems/CollisionSystem';
import WinConditionSystem from './systems/WinConditionSystem';
import { LevelData, VehicleData } from '../types';
export interface GameEngineHandle {
dispatch: (event: object) => void;
}
interface Props {
level: LevelData;
vehicles: VehicleData[];
onWin: () => void;
onMoveCommit: (vehicleId: string, steps: number) => void;
}
const GameEngine = forwardRef<GameEngineHandle, Props>((
{ level, vehicles, onWin, onMoveCommit }, ref
) => {
const engineRef = useRef<any>(null);
useImperativeHandle(ref, () => ({
dispatch: (event) => engineRef.current?.dispatch(event),
}));
const entities = {
grid: { backgroundGrid: level.backgroundGrid },
...Object.fromEntries(vehicles.map(v => [v.id, { ...v }])),
};
return (
<RNGEEngine
ref={engineRef}
systems={[MovementSystem, CollisionSystem, WinConditionSystem]}
entities={entities}
onEvent={(e) => {
if (e.type === 'WIN_TRIGGERED') onWin();
if (e.type === 'MOVE_COMMITTED') onMoveCommit(e.vehicleId, e.steps);
}}
running={true}
/>
);
});
export default GameEngine;

10. Collision Detection — O(1) Grid Map
Collision is handled via a pre-built boolean occupancy map. Each cell is either occupied or empty. Checking if a vehicle
can move N steps requires only N map lookups — all O(1) per cell. The map is rebuilt once per committed move, not
once per frame.
collision.ts — full implementation
// src/utils/collision.ts
import { GRID_SIZE } from './gridUtils';
import { VehicleData } from '../types';
/**
* Returns the number of steps the vehicle can actually move.
* If steps = +2 but only 1 cell is free, returns +1.
* If steps = -3 but 0 cells are free, returns 0.
*/
export function canMove(
vehicle: VehicleData,
requestedSteps: number,
occupancy: boolean[][],
backgroundGrid: number[][],
): number {
const dir = Math.sign(requestedSteps);
const absSteps = Math.abs(requestedSteps);
let allowedSteps = 0;
for (let step = 1; step <= absSteps; step++) {
// Cell at the leading edge of the vehicle after `step` steps
let checkCol: number;
let checkRow: number;
if (vehicle.direction === 'horizontal') {
checkCol = dir > 0
? vehicle.x + vehicle.length - 1 + step // front
: vehicle.x - step; // back
checkRow = vehicle.y;
} else {
checkCol = vehicle.x;
checkRow = dir > 0
? vehicle.y + vehicle.length - 1 + step
: vehicle.y - step;
}
// Out of bounds — hard stop
if (checkCol < 0 || checkCol >= GRID_SIZE ||
checkRow < 0 || checkRow >= GRID_SIZE) break;
// Wall — hard stop
if (backgroundGrid[checkRow][checkCol] === 2) break;
// Another vehicle — hard stop
if (occupancy[checkRow][checkCol]) break;
// Cell is free — allow this step
allowedSteps = step * dir;
}
return allowedSteps;
}
/**
* Returns true if the target vehicle's exit cell is clear.
* Used by WinConditionSystem.
*/
export function isAtExit(
vehicle: VehicleData,
backgroundGrid: number[][],
): boolean {
if (!vehicle.isTarget) return false;
// Exit is always at the right edge for horizontal target
const exitCol = vehicle.x + vehicle.length - 1;
const exitRow = vehicle.y;
// Check if the cell to the right is exit (3) or off-grid

11. Win Condition Logic
WinConditionSystem.ts
// src/engine/systems/WinConditionSystem.ts
import { isAtExit } from '../../utils/collision';
export default function WinConditionSystem(entities: any, { dispatch }: any) {
const target = Object.values(entities).find(
(e: any) => e.isTarget === true
) as any;
if (!target || !entities.grid) return entities;
if (isAtExit(target, entities.grid.backgroundGrid)) {
dispatch({ type: 'WIN_TRIGGERED' });
}
return entities;
}

export function calcStars(
moveCount: number,
thresholds: [number, number, number] // [3star, 2star, 1star]

if (moveCount <= thresholds[0]) return 3;
if (moveCount <= thresholds[1]) return 2;

12. Undo System with MMKV
The undo system stores a stack of MoveRecord objects in Zustand and persists the stack to MMKV after every move.
On app restart the stack is restored so the player can undo moves from a previous session.
gameStore.ts — undo system
// src/store/gameStore.ts
import { create } from 'zustand';

const undoStack = saved ? JSON.parse(saved) : [];
set({ vehicles, undoStack, moveCount: undoStack.length, hintVehicleId: null });
},
commitMove: (record) => {
const { undoStack, vehicles, moveCount } = get();
// Apply move to vehicles array
const updated = vehicles.map(v => {
if (v.id !== record.vehicleId) return v;
return { ...v, x: record.toX, y: record.toY };
});
const newStack = [...undoStack, record].slice(-MAX_UNDO);
storage.set(`undo_${record.vehicleId}`, JSON.stringify(newStack));
set({ vehicles: updated, undoStack: newStack, moveCount: moveCount + 1 });
},
undoMove: () => {
const { undoStack, vehicles } = get();
if (undoStack.length === 0) return;
const last = undoStack[undoStack.length - 1];
const restored = vehicles.map(v => {
if (v.id !== last.vehicleId) return v;
return { ...v, x: last.fromX, y: last.fromY };
});
set({
vehicles: restored,
undoStack: undoStack.slice(0, -1),
});
},
setHint: (id) => set({ hintVehicleId: id }),
}));

13. BFS Hint Engine
The hint engine uses Breadth-First Search to find the shortest solution path from the current board state. It then returns
the first move in that path — the vehicle the player should move next. The same BFS is used offline during level design
to validate solvability.
bfsHint.ts — BFS solver
// src/utils/bfsHint.ts
import { VehicleData, LevelData, MoveRecord } from '../types';
import { buildOccupancyMap } from './gridUtils';
import { canMove, isAtExit } from './collision';
type State = { vehicles: VehicleData[]; path: MoveRecord[] };
function stateKey(vehicles: VehicleData[]): string {
return vehicles.map(v => `${v.id}:${v.x},${v.y}`).sort().join('|');
}
/**
* Returns the next best move (vehicle + direction) or null if unsolvable.
* Timeout: 5000 states explored max to prevent UI freeze on hard levels.
*/
export function bfsSolve(
vehicles: VehicleData[],
level: LevelData,
maxStates = 5000
): MoveRecord | null {
const initial: State = { vehicles: [...vehicles], path: [] };
const queue: State[] = [initial];
const visited = new Set<string>([stateKey(vehicles)]);
while (queue.length > 0 && visited.size < maxStates) {
const current = queue.shift()!;
const target = current.vehicles.find(v => v.isTarget)!;
if (isAtExit(target, level.backgroundGrid)) {
return current.path[0] ?? null; // first move of solution
}
const occupancy = buildOccupancyMap(current.vehicles);
for (const v of current.vehicles) {
for (const dir of [-1, +1]) {
// Try up to 5 steps in each direction
for (let steps = 1; steps <= 5; steps++) {
const allowed = canMove(v, dir * steps, occupancy, level.backgroundGrid);
if (Math.abs(allowed) < steps) break; // blocked before this step
const newVehicles = current.vehicles.map(vv => {
if (vv.id !== v.id) return vv;
return v.direction === 'horizontal'
? { ...v, x: v.x + allowed }
: { ...v, y: v.y + allowed };
});
const key = stateKey(newVehicles);
if (visited.has(key)) continue;
visited.add(key);
const record: MoveRecord = {
vehicleId: v.id,
fromX: v.x, fromY: v.y,
toX: v.direction === 'horizontal' ? v.x + allowed : v.x,
toY: v.direction === 'vertical' ? v.y + allowed : v.y,
};
queue.push({
vehicles: newVehicles,
path: [...current.path, record],
});
}
}
}
}
return null; // no solution found within maxStates

}

progress: Record<number, LevelProgress>;

getStars: (id) => get().progress[id]?.stars ?? 0,

Implementation
React.memo on Vehicle
Prevents re-render of all 5-12 vehicles when only 1 moves

Reanimated worklets on UI thread

All onUpdate logic in Gesture.Pan() is a worklet automatically.

Wrap in React.memo with no props. Use a single View with border lines, no SVG.

Zero async overhead on play — sounds are pre-buffered

~30% JS execution speedup on Hermes vs V8

40-60% size reduction vs PNG with identical visual quality

23. Challenges, Solutions & Edge Cases
Challenge
Root cause
Solution
Gesture conflict — tap on a vehicle behind another
GestureDetector areas overlap when vehicles are adjacent
Use simultaneousWithExternalGesture() to allow parent scroll AND vehicle drag. Each vehicle's GestureDetector is fully independent.
Vehicle teleports on fast swipe
High velocity translationX/Y overshoots grid on low-end devices
Clamp sharedX/Y to grid bounds in onUpdate. Never trust raw translationX — always clamp.
Undo restores wrong position after app restart
Zustand store initialises from JSON but MMKV undo stack may be stale
Store levelId in the MMKV key: undo_{levelId}. Clear stack on new game start.
BFS hint freezes UI on hard levels
BFS explores thousands of states synchronously on JS thread
Run in setTimeout(fn, 0) to yield to UI. Cap at 5000 states. Show spinner on hint button.
Banner ad shifts game content up
Banner inserts height into layout, pushing grid offscreen
Add banner height to gridOffsetY calculation. Use onLayout callback on banner.
Win triggered mid-drag
WinConditionSystem runs every frame including during drag
Add a isDragging flag to the engine. WinConditionSystem skips check when isDragging === true.
Sound plays with 200ms delay on first call
expo-av loads sound lazily on first createAsync call
Preload all sounds at app launch in soundManager.ts and keep Sound objects alive in module scope.
Level JSON fails to parse silently
JSON with a typo (missing comma, wrong key) throws in require() at runtime
Run a build-time JSON validator script (scripts/validateLevels.ts) as a pre-commit hook.
